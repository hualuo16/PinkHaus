# PinkHaus 设计思路文档

> **副标题**：从一句"你觉得这个方案可行吗"开始的产品诞生记
> **日期**：2026-06-29
> **文档性质**：全对话记录提炼的产品设计决策史

---

## 一、缘起：最初的想法

### 1.1 起点

一切从一个问题开始：

> "我想vibe coding一个针对好物测评种草的内容社区app……你觉得这个方案可行吗？"

最初的构想是一个**关联购物平台自动导入购买商品**的测评社区。用户在购物后可以给自己的商品打分（红榜、黄榜、黑榜），还有一个代表超级赞的最高级别叫"pink"，app 名叫 **"pink house"**。

### 1.2 可行性判断

经过分析，这个方案的可行性分布是：

| 部分 | 可行性 | 说明 |
|------|--------|------|
| 粉/红/黄/黑评分体系 | ✅ 轻松可达 | 数据库一个字段就搞定 |
| 内容社区（Feed、搜索、聚合） | ✅ 轻松可达 | 标准 CRUD + 简单聚合查询 |
| 搜索商品看大众评分分布 | ✅ 轻松可达 | 按商品 ID group by 评分等级 count |
| 关联购物平台自动导入 | ⚠️ 需要取舍 | 无公开 API，原型阶段建议改用手动添加 |

**核心决策**：先做纯前端原型验证产品逻辑，不做后端，数据存 localStorage。

---

## 二、第一次聚焦：缩小范围

### 2.1 用户提议的修正

快速砍掉了"关联购物平台"这个重功能，聚焦于：

- **只做化妆品和护肤品**
- **用户手动添加商品**，不上传图片（降低发帖门槛）
- **用户输入商品名/品牌名 → AI 自动匹配**我们内置的商品库
- **支持语音输入**
- **肤质强制填写**，展示评价时醒目显示
- **同一商品下的所有评价自动聚合**

### 2.2 关于产品图片的决策

> "用户无需上传商品图因为很麻烦大家不愿意做"

所以决定：**不上传图片**，所有商品默认由系统生成粉色系占位图。这个设计后来演进为品类商品页的粉色背景 + 品牌首字母展示风格。

## 三、产品命名与品牌

### 3.1 名字演变

| 阶段 | 名称 | 理由 |
|------|------|------|
| 初始 | Pink House | 粉色小屋，温暖可爱的社区感 |
| 中期 | PinkHaus | 改为德语拼写，"高级感"上升 |

### 3.2 品牌视觉规则

最终确定的品牌展示规范：

- **Pink** → 粉色 (#F27B9D)，加粗
- **Haus** → 黑色 (#2C2C2C)，加粗
- P 和 H 大写，中间无空格
- 字体：PinkHaus 标题使用胖胖体风格

---

## 四、评分体系：粉红黄黑四档

### 4.1 评级定义

| 等级 | 标签 | 含义 | 色值 |
|------|------|------|------|
| 💖 PINK | 超级赞 | 必买清单，无限回购 | #F27B9D |
| ❤️ 红榜 | 推荐 | 值得入手，整体满意 | #FF6B6B |
| 💛 黄榜 | 一般 | 无功无过，可有可无 | #E8A800 |
| 🖤 黑榜 | 不推荐 | 踩雷，不会回购 | #555 |

### 4.2 设计考量

这个评级体系的核心设计思路是：

1. **PINK 作为最高级**：与品牌名 PinkHaus 呼应，是社区的最高荣誉
2. **红榜/黑榜**：借用中文互联网已有的消费评价语境（"上红榜/黑榜"）
3. **黄榜**：给"还行但不惊艳"的产品一个位置，避免非黑即白的评价压力
4. **数量而非分数**：不设平均分，而是展示各等级的人数，更直观

## 五、肤质体系

### 5.1 肤质是核心维度

从第一天就确定：**肤质是 PinkHaus 的核心差异化**。所有用户必须填写肤质，所有评价旁都要醒目展示评价者的肤质。

### 5.2 肤质选项的迭代

**第一版（直白五类）：**

| 干性 | 油性 | 混合性 | 中性 | 敏感性 |

**第二版（混合性细分）：**

| 干性 | 油性 | 混合-偏油 | 混合-偏干 | 中性 | 敏感性 |

> "选择肤质的时候 混合皮点进去再分成混合皮-偏油和混合皮-偏干两个选择"

**交互设计**：
- 主选项显示"混合性"
- 点击后展开子选项：「混合-偏油」「混合-偏干」
- 选中子选项后，主界面显示"混合性"卡片保持选中态，下方显示"已选：混合-偏油"
- 再次点击"混合性"可重新选择子选项

### 5.3 肤质显示规则

- 社区 Feed 流每条评价旁显示肤质标签（如「🌵 干皮」「🌓 混油」）
- 商品详情页评价列表同理
- 个人主页展示肤质，可随时编辑

## 六、注册流程：从一页到多步递进

### 6.1 初始设计

最初所有字段放在一个页面上：昵称 + 肤质选择 + 提交。简洁但缺少仪式感。

### 6.2 多步递进式注册

应要求改为**分步引导**，每一步只让用户做一件事：

`
Step 0：欢迎页（热情介绍 PinkHaus）
Step 1：输入昵称
Step 2：选择性别
Step 3：肤质（测试 or 手动选择）
Step 4：（可选）个性签名
Step 5：完成，进入 app
`

### 6.3 欢迎页文案

原始版本比较正式（"这里是一个只聊美妆护肤的真实测评社区"），被评价为"AI 味太重"后改为：

> 姐妹们！你终于来了！🎀
> 这里是 PinkHaus — 一个只聊真实美妆护肤的宝藏小窝
> 没有广告、没有恰饭、只有姐妹们用过之后的真心话
> 想知道某个精华到底值不值得买？
> 想找同肤质的姐妹抄作业？
> 来这里就对了 ✨
> 只需要几步就能进来玩啦～

### 6.4 肤质测试

到肤质那步时，先问用户：「想不想做个简单的肤质测试？」vs「我知道自己的肤质，跳过测试」

测试为 3 道题（Baumann 风格简化版）：
1. 洗脸后多久出油？
2. 脸颊毛孔状况？
3. 长痘痘的频率？

根据回答自动判断：油性 / 混合性 / 干性。若结果为混合性，则弹出子选项选偏油/偏干。

## 七、搜索与品类浏览

### 7.1 搜索功能

支持两种搜索方式：

1. **文字搜索**：输入商品名或品牌名，模糊匹配内置产品库
2. **语音搜索**：基于 Web Speech API，点击麦克风按钮语音输入

### 7.2 品类图标评价页

评价页的品类选择经历了两轮迭代：

**第一版**：小巧的 tag chips（标签式）

**第二版**：大图标，2 列 3 行 + 其他整行

`
┌──────────┐  ┌──────────┐
│   🧴     │  │   💧     │
│   面霜    │  │   水乳    │
├──────────┤  ├──────────┤
│   👁️     │  │   🎭     │
│   眼霜    │  │   面膜    │
├──────────┤  ├──────────┤
│   ✨     │  │   💋     │
│   精华    │  │   唇部    │
├────────────────────────┤
│         📦 其他         │
└────────────────────────┘
`

每个品类图标：
- 点击后跳转到**独立的品类商品选择页**
- 该页展示该品类下所有商品
- 每款商品有粉色系占位图 + 品牌首字母 + 商品名
- 参考小红书笔记风格，但**所有卡片等高**（aspect-ratio: 1 保证正方形）

## 八、个人主页

### 8.1 小红书风格

个人主页的设计灵感明显来自小红书：

**结构：**
1. 封面背景图（高度 140px，可切换 9 种系统样式）
2. 圆形头像 + 昵称 + 性别 + 肤质标签（部分覆盖在背景图上）
3. 个性签名（点击即编辑）
4. 三个 Tab：「我的评价 (N)」「收藏商品 (N)」「收藏评价 (N)」

### 8.2 封面背景

提供 9 种系统背景样式：

| 类型 | 款式 |
|------|------|
| 纯色 | 蜜桃粉、薰衣草、薄荷、天空蓝、蜜桃橙 |
| 条纹 | 粉条纹、紫条纹 |
| 波点 | 粉波点、绿波点 |

> "粉色放最前面推荐"

背景选择在个人主页右上角 🎨 按钮触发，直接在当前页切换，不跳转。

### 8.3 收藏系统

- 用户可以**收藏商品**（在商品详情页操作）
- 也可以**收藏其他用户的评价**
- 收藏内容在个人主页的 Tab 中查看

## 九、UI 风格

### 9.1 配色方案

> "统一ui为饱和度稍微低一点的蜜桃粉"

| 色值 | 用途 |
|------|------|
| #F27B9D | 主色——低饱和度蜜桃粉 |
| #FFF0F4 | 浅粉色背景 |
| #D96A89 | 按 hover |
| #2C2C2C | 正文黑色 |
| #999 | 辅助灰色 |
| #F0E8E6 | 边框 |

### 9.2 图标风格

> "所有小图标都做成简约版只要大致的形状和填色的色块"

导航栏图标从 emoji 改为 **SVG 简约填充图标**：
- 首页：房子形状
- 搜索：放大镜
- 评价：铅笔
- 我的：人物轮廓

导航栏颜色规则：
- 未选中 → 灰色
- 选中 → 蜜桃粉

### 9.3 字体

页面正文使用系统字体。PinkHaus 标题使用了 Google Fonts 的 ZCOOL KuaiLe（胖胖体），符合"胖胖腿一样的字体"的要求。

## 十、品类商品选择页

### 10.1 页面设计

点击品类图标后跳转到独立的商品选择页，设计参考了小红书笔记页：

**样式要点：**
- 2 列网格布局
- 每个商品一张粉色系背景图（非真实产品图）
- 图片中央展示品牌+商品的首字母缩写
- 下方展示品牌名（粗体）+ 商品名
- **所有卡片等高等宽**，不似小红书参差不齐

### 10.2 占位图生成

由于用户不上传图片，系统根据商品 ID 的哈希值从 8 种粉色系中选取一个作为背景色，生成 SVG 占位图：

`
色调池：['#F8D7DA', '#F0C8D0', '#FFE0E8', '#FFD6E0',
         '#FADADD', '#F5D0D8', '#FFE4EC', '#F8E0E8']
`

## 十一、技术架构决策

### 11.1 技术选型

| 决策 | 选择 | 理由 |
|------|------|------|
| 框架 | React 18 | 生态成熟，组件化，HMR 开发体验好 |
| 构建 | Vite 5 | 秒级冷启动，ESM 原生 |
| 路由 | React Router v6 | SPA 标配 |
| 数据 | localStorage | 纯前端原型，零后端成本 |
| 包管理 | pnpm | 快速，磁盘效率高 |
| 语音 | Web Speech API | 浏览器原生，无额外依赖 |

### 11.2 为什么不走后端

在原型阶段，所有数据存在 localStorage 中。这意味着：
- **优点**：开箱即用，无需部署，无需数据库
- **缺点**：数据仅限本地，无法多用户在线协作

路线图中将后端 + 数据库列为 P0 优先级。

## 十三、设计过程中的关键决策记录

### 决策 1：不做购物平台导入
原始想法中有"关联购物平台自动导入购买商品"，但分析后发现没有公开 API，原型阶段改为**用户手动添加 + AI 匹配商品库**。

### 决策 2：不上传商品图片
认为用户不愿意为每件商品找图，决定由系统**自动生成粉色占位图**。这个限制反而成了 PinkHaus 视觉风格的一部分。

### 决策 3：混合性肤质要细分
> "混合皮点进去再分成混合皮-偏油和混合皮-偏干"

这个细节是在用户试用后提出的，说明肤质的精准度对评测参考价值很重要。

### 决策 4：品类图标从标签改大图标
> "评价页面增加的品类不是指小标签，我要大图标，两个一排一页3排填满空间"

从 tag chips 到 2 列大图标，交互范式的转换体现了"可视化浏览 > 文字标签"的设计偏好。

### 决策 5：品类页独立跳转
品类图标从原本的 inline 搜索改为跳转独立页面，借鉴了小红书笔记页面的**图文网格**布局，但强制所有卡片等高（避免了小红书参差不齐的风格）。

### 决策 6：Tab 去掉"上面那一栏"
个人主页精简了顶部统计栏，将数量信息直接整合到 Tab 标签中（"我的评价 (N)"），去掉了重复内容。

### 决策 7：返回导航统一用 navigate(-1)
品类页的返回按钮从 
avigate('/review') 改为 
avigate(-1)，解决了"评价页点返回又回到品类页"的导航死循环。

## 十四、关于"vibe coding"的实践

这次开发过程本身就是一次**完整的 vibe coding 实践**：

1. **从想法开始**：不是先写 PRD 再实现，而是先试再说
2. **渐进式细化**：用户每试用一次，就提出更精准的改进
3. **跳过基建直接做交互**：没有后端、没有数据库、没有 CI/CD，第一天就跑起来了
4. **以假乱真的数据**：25 条示例评价 + 150+ 商品库，一打开就能看到活的内容
5. **在浏览器里调试产品**：不是在代码里调逻辑，而是在真实 UI 上感受"这里不对"

整个过程从第一行代码到可以交互的原型，跑在 http://localhost:5173 到 5182 的不断迭代中，见证了产品从模糊想法到可触摸形态的生长过程。
